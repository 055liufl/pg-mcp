# 结果评分规则(scoring.md)

> 当 SQL 顺利执行后,skill 必须自评结果是否「有意义」并在 0-10 给出置信分。**< 7 分需要重新生成 SQL 并重试**。本文件给出可执行的评分清单。

## 1. 输入

每次评分接收三件输入:
- `user_intent` — 用户的自然语言原始问题
- `generated_sql` — 当前生成并执行成功的 SQL
- `result_sample` — psql 返回的前 N 行(N ≤ 10)+ 总行数 + 列名

## 2. 评分维度(每项 0-2 分,总分 10)

| 维度 | 0 分(完全失败) | 1 分(部分对) | 2 分(完美) |
|---|---|---|---|
| **维度 A:意图覆盖** | 没回答用户问题(返回的列与问题无关,例如问 "Top 客户" 却给了一堆订单号) | 部分回答(返回了相关数据但不完整,例如问 "近 30 天" 但 SQL 里没有日期过滤) | 完全回答用户的问题 |
| **维度 B:列正确** | 关键列缺失或错误(例如问 "总收入" 却返回 `count(*)`) | 关键列在,但有冗余无关列拖累可读性 | 列恰好对应问题,顺序合理 |
| **维度 C:过滤/聚合正确** | WHERE/GROUP BY/HAVING 漏了用户隐含的条件(如 status 过滤) | 大致对,但边界值有歧义(如 "近 30 天" 用了 `> 30` 而不是 `>=`) | 过滤与聚合精确匹配语义 |
| **维度 D:行数合理** | 0 行(空结果且用户问题应当有数据)、或 LIMIT=100 截断了应有的少量结果 | 行数偏多/偏少但不至于让用户误解 | 行数与问题预期吻合;若是 Top-N 类问题,LIMIT 与 N 一致 |
| **维度 E:数据"看起来对"** | 数值明显反常(收入为 0、负数、空字符串、全 NULL、所有行 metric 相同) | 数值合理但有少量可疑(如某行 NULL 在不应该 NULL 的列) | 数值通过常识检查 |

**总分** = A + B + C + D + E。最高 10。

## 3. 自动否决项(直接 0 分)

无论上面五维拿到多少,只要触发以下任一,**总分判为 0,必须重试**:

1. SQL 抛错(skill 在执行阶段已检测,这里属于回退)
2. 返回 0 行,但用户问题语义上预期 ≥1 行(例如 "数据库里有几个用户?")
3. 关键列(如金额、客户、商品)全部为 `null`
4. 维度 C 错(把 `published` 写成 `draft` 之类的拼写错误)
5. 数据库与用户语义错配(例如把博客问题路由到 analytics_dw)

## 4. 阈值

| 总分 | 行动 |
|---|---|
| **9 - 10** | 高置信,直接返回 |
| **7 - 8** | 接受,但在最终回答里加一句简短的不确定性说明 |
| **4 - 6** | **不接受**。必须重试:
  - 先检查路由(可能选错数据库)
  - 再检查 SQL 中过滤/聚合语义,深度思考用户原意
  - 重写 SQL,回到执行阶段 |
| **0 - 3** | 严重错误。必须重试。同时考虑用户问题是否本身有歧义,**两次连续 < 4** 应当返回澄清问题给用户而不是再尝试 |

## 5. 重试预算

- 每次用户请求最多 **3 次** 重试(也就是最多 4 次执行)。
- 第 4 次仍 < 7,不要再重试。返回当前最佳答案 + 一段坦诚的「以下结果置信度仅 X/10,因为...」说明。

## 6. 评分时常见的 trap

下列情况容易让 LLM 给虚高分,**评分时务必当心**:

1. **SQL 没报错但语义错** —— SQL 跑通不等于答对题目。例:用户问 "上月 GMV",SQL 用 `placed_at >= NOW() - INTERVAL '30 days'`(滑动 30 天),不是月份。这个 D 维度是 1 分,C 维度是 1 分。
2. **聚合粒度错** —— 用户问 "每月",SQL 写的是 "总和"。结果只有 1 行,D 维度 0 分。
3. **重复行** —— 多对多 JOIN 没 DISTINCT 导致重复,数值翻倍。E 维度看到金额诡异地大就要警惕。
4. **NULL 没处理** —— `SUM(NULL) = NULL`,聚合结果 NULL。E 维度 0 分。
5. **大小写/枚举值** —— `WHERE status = 'Published'` 在严格枚举上为 0 行,实际值是 `'published'`。**必须查 reference**。
6. **物化视图过期** —— 在 `analytics_dw` 里用 `reporting.monthly_revenue_mv`,如果用户问的是 "今天" 的收入,MV 可能滞后,得改用 fact 现算。
7. **跨表 JOIN 用业务键而非代理键** —— SCD-2 维度表里用 `customer_bk` JOIN 会出现重复历史版本。

## 7. 自评模板(每次必填)

```yaml
intent_summary: <一句话复述用户意图>
db_chosen: <mini_blog | shop_oms | analytics_dw>
sql_summary: <一句话描述 SQL 思路>
sample_rows_inspected: <数字,通常 5-10>
total_rows_returned: <数字>

scores:
  A_intent_coverage:   <0|1|2> # 是否覆盖用户问题
  B_columns_correct:   <0|1|2>
  C_filter_aggregate:  <0|1|2>
  D_row_count_sane:    <0|1|2>
  E_data_looks_right:  <0|1|2>
  total: <0..10>

verdict:
  decision: <accept | retry | give_up>
  reason: <短句>
  next_action: <如果 retry,要改什么>
```
